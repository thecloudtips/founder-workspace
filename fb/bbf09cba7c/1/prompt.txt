# Writing Plans

## Overview

Write comprehensive implementation plans assuming the engineer has zero context for our codebase and questionable taste. Document everything they need to know: which files to touch for each task, code, testing, docs they might need to check, how to test it. Give them the whole plan as bite-sized tasks. DRY. YAGNI. TDD. Frequent commits.

Assume they are a skilled developer, but know almost nothing about our toolset or problem domain. Assume they don't know good test design very well.

**Announce at start:** "I'm using the writing-plans skill to create the implementation plan."

**Context:** This should be run in a dedicated worktree (created by brainstorming skill).

**Save plans to:** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- (User preferences for plan location override this default)

## Scope Check

If the spec covers multiple independent subsystems, it should have been broken into sub-project specs during brainstorming. If it wasn't, suggest breaking this into separate plans — one per subsystem. Each plan should produce working, testable software on its own.

## File Structure

Before defining tasks, map out which files will be created or modified and what each one is responsible for. This is where decomposition decisions get locked in.

- Design units with clear boundaries and well-defined interfaces. Each file should have one clear responsibility.
- You reason best about code you can hold in context at once, and your edits are more reliable when files are focused. Prefer smaller, focused files over large ones that do too much.
- Files that change together should live together. Split by responsibility, not by technical layer.
- In existing codebases, follow established patterns. If the codebase uses large files, don't unilaterally restructure - but if a file you're modifying has grown unwieldy, including a split in the plan is reasonable.

This structure informs the task decomposition. Each task should produce self-contained changes that make sense independently.

## Bite-Sized Task Granularity

**Each step is one action (2-5 minutes):**
- "Write the failing test" - step
- "Run it to make sure it fails" - step
- "Implement the minimal code to make the test pass" - step
- "Run the tests and make sure they pass" - step
- "Commit" - step

````markdown
    assert result == expected
```

- [ ] **Step 2: Run test to verify it fails**

Expected: FAIL with "function not defined"

- [ ] **Step 3: Write minimal implementation**

```python
def function(input):
    return expected
```

- [ ] **Step 4: Run test to verify it passes**

Run: `pytest tests/path/test.py::test_name -v`
Expected: PASS

- [ ] **Step 5: Commit**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

   - Repeat until ✅ Approved

**Chunk boundaries:** Use `## Chunk N: <name>` headings to delimit chunks. Each chunk should be ≤1000 lines and logically self-contained.

## Critical Issues

**1. `node -c` syntax check will fail on ESM files — wrong verification command (Confidence: 90)**

Tasks 1 and 2 both use `node -c <file>.mjs` as the verification step. However, `node -c` does not handle ES module syntax (`import`/`export`) correctly on many Node.js versions — it treats the file as CommonJS and throws `SyntaxError: Cannot use import statement in a module` rather than validating it. The correct approach is `node --input-type=module --eval "$(cat <file>)"` or simply `node <file> --help` / importing the module in a test script. Using `node -c` against a live API key is also impossible, so the `--validate-only` mode is the right CI-safe check.

- Plan line 231: `Run: node -c _infrastructure/auth/resolve-env.mjs`
- Plan line 541: `node -c scripts/late-tool.mjs`

Fix: Replace both verification steps with a non-executing import check, e.g.:
```bash
node --input-type=module --eval "import './_infrastructure/auth/resolve-env.mjs'"
```
or rely on the `--validate-only` path for `late-tool.mjs`.

**2. `resolve-env.mjs` does not validate key format (no `LATE_API_KEY` length check) — spec requires `sk_` + 64 hex characters (Confidence: 85)**

The spec states: "API key format: `sk_` prefix + 64 hex characters." The `resolveKey()` function in the plan only checks that the key starts with the provided `prefix`. There is no validation that the remainder is exactly 64 hex characters. The spec says "Validate key format before making API calls (fail fast on malformed keys)." The plan's `resolveKey` only does prefix checking, which is insufficient — `sk_` alone would pass.

**3. `late-tool.mjs` leaks the raw API key in the 401 error message (Confidence: 95)**

At plan line 324–327, the 401 error message calls `getApiKey()` a second time to pass into `maskKey()`:
```javascript
throw Object.assign(
  new Error(`Authentication failed. Verify your LATE_API_KEY (${maskKey(getApiKey())})`),
  { code: 'LATE_AUTH_FAILED' }
);
```

`getApiKey()` calls `resolveKey()` which re-reads the key from the environment. This is functionally correct — it produces a masked string. However, the spec mandates "NEVER log, echo, or include the API key in error messages." Because `maskKey()` is used, this is technically compliant. That said, there is a subtle risk: if `maskKey()` itself throws (e.g., key is undefined at that point due to a race), the unmasked key could surface in the exception message's stack. A safer pattern used in `notion-tool.mjs` is to resolve the key once and keep a reference. This is a low-severity concern but worth noting since the spec has an explicit "NEVER" rule.

Fix: Resolve the key once at the top of `apiCall`, store in a local `const`, and use that reference for masking in the error.
